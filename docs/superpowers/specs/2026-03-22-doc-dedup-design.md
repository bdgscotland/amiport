## Problem Statement

Project instructions are duplicated across CLAUDE.md (315 lines), `.claude/rules/`, `.claude/agents/`, `.claude/skills/`, and `docs/`. When one copy is updated, the others drift. The user (Duncan) has to manually prompt Claude to update docs. There is no automated enforcement for content consistency — only name-level checks (`check-docs` validates agent names appear in docs, nothing more).

Claude Code docs recommend CLAUDE.md be under 200 lines for best adherence. Ours is 315+ lines, with ~155 lines duplicated from rules files that already load via path-scoping.

## Solution

Two-part fix:

1. **Trim CLAUDE.md to ~150 lines** — remove content that's duplicated in path-scoped rules files. Rules files become the canonical sources. CLAUDE.md becomes a concise project overview.

2. **Enhance `check-docs` validation** — add 4 new cross-reference checks that catch drift between agents, skills, shim functions, and their documentation references. Wire into pre-commit + CI (already wired — just extending the script).

## Part 3: Enhanced Validation

Extract existing inline `check-docs` from Makefile into `scripts/check-docs.sh`. Add 4 new checks:

### Check 1: Agent cross-reference
- Every `.claude/agents/*.md` → must have a row in CLAUDE.md's agent table AND in `rules/use-pipeline-agents.md` dispatch table
- Every agent in `rules/use-pipeline-agents.md` → must have matching `.claude/agents/*.md` file
- **Prerequisite:** Before this check lands, add missing agents (debug-agent, hardware-expert, memory-checker) to `rules/use-pipeline-agents.md`'s dispatch table
- *(Existing check already validates CLAUDE.md + README + architecture.md — extend to include use-pipeline-agents.md)*

### Check 2: Skill existence
- Every `.claude/skills/*/SKILL.md` with a top-level slash command (invoked via `/skill-name`) → must appear in CLAUDE.md
- **Exclude sub-skills** that are only called by other skills/agents (e.g., `write-arexx` is called by the test pipeline, not directly by users). Detection: if a skill's `SKILL.md` contains `argument-hint:` or is referenced in another skill's prompt as a sub-skill, it's user-facing. Otherwise, check CLAUDE.md.

### Check 3: Shim function sync
- Only POSIX-replacement functions (those with a direct POSIX equivalent: `amiport_open` → `open`, `amiport_fopen` → `fopen`, etc.) must appear in `.claude/skills/transform-source/references/transformation-rules.md`
- **Exclude internal helpers** that have no POSIX equivalent: `amiport_errno_from_ioerr`, `amiport_expand_argv`, `amiport_check_break`, `amiport_fd_is_valid`, `amiport_fd_to_fh`, `amiport_exit`, etc.
- **Detection heuristic:** function name must match `amiport_<posix_name>` where `<posix_name>` is a known POSIX function. Maintain an exclusion list for helpers.
- Catches: new shim function added via `/extend-shim` but transform-source doesn't know about it

### Check 4: Duplication regression guard
- CLAUDE.md must NOT contain marker strings from canonical rule files:
  - `fopen() macro collision` (known-pitfalls.md content)
  - `vamos argv pointer` (known-pitfalls.md content)
  - `vsnprintf(NULL` (known-pitfalls.md content)
  - `pledge/unveil` (known-pitfalls.md content)
  - `ANSI C89` as a heading/rule (amiga-coding.md content)
- Simple grep guard. If duplicated content creeps back into CLAUDE.md, the commit is blocked.

### Integration
- `scripts/check-docs.sh` replaces the inline Makefile target
- Pre-commit hook already runs `make check-docs` — no change needed
- CI already runs `make check-docs` — no change needed

## Decisions Made

1. **Trim aggressively** — CLAUDE.md goes from 315 to ~150 lines
2. **Rules files are canonical** — they already exist and are path-scoped
3. **No @imports needed** — rules load automatically; @imports would re-add the context cost
4. **Duplication regression guard** — prevents content drifting back into CLAUDE.md
5. **Extend check-docs, not new script** — keeps the existing integration points

## Risk Mitigation

The biggest risk is trimming CLAUDE.md causes Claude to miss important context. Mitigated by:

1. **Path-scoped rules already load** when editing port/lib code (amiga-coding.md, known-pitfalls.md)
2. **Global rules load every session** (use-pipeline-agents, documentation, test-hygiene, port-directory-hygiene)
3. **Duplication regression check** prevents content creeping back
4. **Incremental approach** — trim and validate before committing; run a test conversation to verify Claude still follows the rules
5. **Path-scoping asymmetry is low-risk** — `shim-usage.md` is path-scoped to `lib/**` but someone editing `ports/**` might need it. Mitigated by CLAUDE.md's Key References section (kept) which points to shim docs, and by the fact that port editing almost always involves reading shim headers (which triggers `lib/**` rules)
