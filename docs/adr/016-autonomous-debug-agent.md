# ADR-016: Autonomous Debug Agent with Enforcer Integration

## Status

Accepted

## Date

2026-03-21

## Context

Ported programs crash on AmigaOS with no diagnostic output. When a port triggers a Guru Meditation or illegal memory access inside FS-UAE, Claude gets nothing actionable — just a dead emulator. The existing FS-UAE testing pipeline (ADR-014) handles the happy path (run tests, capture TAP results, exit cleanly), but when things go wrong, there is a black hole.

Manual debugging requires Amiga expertise: boot FS-UAE, reproduce the crash, run Enforcer, read the hit report, cross-reference the address with a linker map, find the bug, fix it, and repeat. This is the most time-consuming and expertise-dependent part of porting. It breaks the autonomous porting loop — every crash becomes a human bottleneck.

Two approaches were considered for interactive debugging:

1. **Enforcer pipeline only** — passive crash detection via serial, map addresses to source, fix and retest. No interactive debugger. Covers the common cases (NULL derefs, use-after-free, stack overflow) but cannot inspect runtime state at crash site.
2. **Enforcer + bgdbserver** — everything from (1), plus on-demand GDB sessions via bgdbserver for deeper investigation when passive detection is insufficient.
3. **Enforcer + uae-dap** — everything from (1), plus a patched FS-UAE with embedded GDB server. Most powerful but requires maintaining custom emulator builds.

## Decision

### Two-layer architecture: Layer 1 ships now, Layer 2 deferred

**Layer 1 (Enforcer passive detection, always-on)** ships immediately and provides the complete crash detection, source mapping, and autonomous fix loop for common crash patterns.

**Layer 2 (bgdbserver/GDB, on-demand)** is deferred until TCP routing through FS-UAE's bsdsocket emulation is empirically verified. Layer 1 alone covers the overwhelmingly common crash patterns.

### Layer 1: Passive Crash Detection

#### FS-UAE serial-to-TCP for Enforcer output capture

Debug mode is activated via `test-fsemu.sh --debug`. The existing `generate_config()` function is extended to add debug-specific settings:

- `cpu = 68030` — enables MMU for Enforcer (differs from stock 68020 test runs)
- `serial_port = tcp://127.0.0.1:<PORT>/wait` — serial output to TCP with ephemeral OS-assigned port
- `jit_compiler = 0` — JIT interferes with debugging
- `bsdsocket_library = 1` — required for Layer 2 (off by default in FS-UAE)

The serial port uses an **ephemeral OS-assigned port** (not hardcoded). `serial-capture.py` binds port 0, the OS assigns a free port, and the actual port number is written to a sentinel file. This eliminates port conflicts from stale processes or concurrent debug sessions. The `/wait` flag pauses Amiga boot until the host serial listener connects, ensuring no Enforcer output is lost.

Debug mode uses a 90-second timeout (3x normal) to account for MMU overhead and serial startup delay.

#### Amiga-side debug tools

SegTracker, Enforcer, and Mungwall are loaded in the Startup-Sequence before the test harness. SegTracker must start before Enforcer (Enforcer's `STACKCHECK` option depends on it). Both Enforcer and Mungwall output to the serial port by default — no special flags required.

#### STABS debug info in all builds

All builds include `-gstabs` (the only debug flag recommended for AmigaOS with the bebbo toolchain). Binaries are stripped only at package time (`make package`). This ensures every committed binary is always debuggable. The bebbo toolchain outputs Amiga hunk format directly (no ELF intermediate); STABS info is stored in HUNK_DEBUG blocks readable by `m68k-amigaos-addr2line`.

#### addr2line with `--section=.text` for source mapping

`debug-report.py map` takes hunk number + offset from each Enforcer hit and maps it to source file and line via `m68k-amigaos-addr2line -e <binary> --section=.text <address>`. For multi-hunk programs, the linker map file (generated by `-Wl,-Map`) determines each hunk's base address. Multi-hunk binaries without a `.map` file produce an explicit error rather than silently guessing wrong. If addr2line returns `??:0`, the report indicates "no debug info — rebuild with -gstabs."

#### Printf breadcrumbs for hard crash diagnosis

Some crashes (illegal instruction, bus error on non-MMU-protected address) trigger a Guru Meditation that resets the machine before Enforcer can report. These are detected by absence: FS-UAE exits without UAEQuit, enforcer.log is empty, and TAP results show no test completion. The debug agent handles hard crashes by inserting `fprintf(stderr, "CHECKPOINT: function_name\n")` at function entry points — stderr goes to the serial port (already captured). The last checkpoint before the crash narrows the location to a specific function, then finer-grained checkpoints are inserted and the test is rerun.

#### Git stash for rollback safety

Before each fix attempt, the debug agent creates a named git stash (`git stash push -m 'debug-attempt-N'`). If a fix introduces a regression (new crash at a different location, or more Enforcer hits than before), the agent pops the stash and tries a different approach. Named stashes are visible via `git stash list`, discoverable if the agent is interrupted, and survive reboots. Maximum 5 fix-rebuild-retest iterations before escalating to the user.

#### Crash pattern knowledge base

A persistent knowledge base at `docs/references/crash-patterns.md` stores structured crash signature to fix pattern entries. When the debug agent fixes a crash, it appends an entry. Over time this becomes a lookup table that accelerates diagnosis — the agent checks the KB before reasoning from scratch.

### Layer 2: On-Demand GDB Debugging (Deferred)

bgdbserver listens on TCP inside the emulated Amiga. FS-UAE's bsdsocket emulation uses real host sockets, so the Amiga-side TCP port should be accessible from the host. However, this needs empirical verification before implementation.

**Verification gate:** Run bgdbserver inside FS-UAE listening on a TCP port, then `telnet localhost <port>` from the host. Layer 2 implementation is blocked until this passes.

If TCP routing fails, the alternative is `astub` (a serial-based GDB stub), not routing bgdbserver over serial — bgdbserver is TCP-only.

## Consequences

### Positive

- All ports are debuggable without Amiga expertise — Claude autonomously detects, diagnoses, and fixes crashes
- The Enforcer pipeline is transparent to existing ports — no changes required to use it
- Crash pattern knowledge base compounds across ports, accelerating future diagnosis
- STABS debug info in all committed binaries means any binary can be debugged at any time
- Git stash rollback prevents the debug loop from making things worse
- Layer 1 alone covers the most common crash patterns (NULL derefs, use-after-free, stack overflow)

### Negative

- 68030 CPU required for debug runs (Enforcer needs the MMU) — differs from stock 68020 test runs. The 68030 data cache could theoretically mask or expose timing-sensitive bugs, but this is acceptable since debug mode targets crashes, not performance
- Binaries are slightly larger with STABS info (a few KB for typical CLI ports). Stripped at package time for Aminet distribution
- Debug mode is slower than normal testing (90s timeout, MMU overhead, serial I/O)
- Layer 2 (GDB) deferred — subtle crashes that need runtime state inspection require source-level reasoning until bgdbserver TCP routing is verified

### Neutral

- Layer 2 can be implemented independently when the verification gate passes — no rework of Layer 1 required
- Debug tools (Enforcer, Mungwall, SegTracker) are freeware from Aminet — no licensing concerns
- The tooling built here (serial capture, Enforcer parser, addr2line integration) is useful beyond amiport for the broader Amiga/retro computing community
