## Vision

### 10x Check

The current plan converts 6,700 HTML pages to markdown. The 10x version makes amiport the **definitive machine-readable AmigaOS reference** — not flat files, but a knowledge graph that AI agents can reason with. Every function linked to its library, parameters, structs, prose explanations, code examples, and the POSIX functions it replaces. An agent porting a program wouldn't just mechanically swap function calls — it would understand the *why* behind each AmigaOS API pattern.

### Platonic Ideal

It would feel like having **a senior Amiga developer pair-programming with you**. You'd say "port this program that uses POSIX file locking" and the system wouldn't just mechanically replace `flock()` with a stub. It would know that AmigaOS uses cooperative file locking via `Lock()/UnLock()`, that there's no byte-range locking, that shared/exclusive semantics differ, and it would point you to the exact chapter in the Libraries Manual that explains the difference — with example code from Amiga Mail that demonstrates the pattern.

## Accepted Scope — Specifications

### Expansion 1: Porting Summaries

**What:** Each chapter's `_index.md` gets a `porting_summary` field in its YAML frontmatter.

**Schema:**
```yaml
porting_summary:
  relevance: HIGH | MEDIUM | LOW
  summary: "2-3 sentence description of why this chapter matters for porting"
  posix_equivalents: [malloc, calloc, realloc, free]  # POSIX functions this replaces
```

**How determined:** Heuristic in Pass 2 — match chapter content against the POSIX-to-AmigaOS mapping from `posix-to-amiga-map.md`. Chapters mentioning many mapped functions = HIGH. Chapters about GUI/hardware with no POSIX equivalents = LOW. Summary text is template-generated from detected function names and chapter title.

**Generated in:** Pass 2 enrichment.

### Expansion 2: Include File → Docs Mapping

**What:** `docs/references/adcd/INCLUDES.json` mapping Amiga include paths to ADCD chapters.

**Schema:**
```json
{
  "proto/dos.h": {
    "library": "dos.library",
    "chapters": ["libraries/27-dos-library/", "libraries/28-dos-library-pt2/"],
    "autodoc": "autodocs/dos.library.md"
  }
}
```

**How discovered:** Derive from chapter titles containing library names (e.g., "DOS Library" → `proto/dos.h`). Cross-reference with the known library list from `parse-autodocs.py`'s FETCH_LIBRARIES. Manual corrections for non-obvious mappings (e.g., "BOOPSI" → `proto/intuition.h`).

**Generated in:** Pass 2 enrichment.

### Expansion 3: Example Code Extraction

**What:** Extract code examples from all manuals into standalone `.c` files at `docs/references/adcd/examples/{library}/`.

**Detection heuristic:** During Pass 2, identify code blocks (already detected in Pass 1 by the 4+ space indentation + code character heuristic). Filter to blocks that:
- Contain at least one function call (`FuncName(`)
- Are 5+ lines long (skip one-liners like function signatures)
- Contain `#include` or `{` or `return` (likely a complete-ish example)

**Output format per file:**
```c
/* Source: Amiga RKM Libraries, Chapter 20: Exec Memory Allocation
 * Section: Allocating System Memory
 * Demonstrates: AllocMem/FreeMem usage with MEMF_PUBLIC
 * Libraries: exec.library
 * ADCD reference: libraries/20-exec-memory/allocating-system-memory.md
 */

[extracted code]
```

**Naming:** `{library}/{slugified-section-title}.c`. Collision → append `-2`.

**Note:** This supersedes the design doc's Open #2 decision to "start with inline." The CEO review determined that extracting examples has clear agent value and the marginal cost is low (the code block detection already exists from Pass 1).

### Expansion 4: Autodocs Version Diff (CONDITIONAL)

**What:** `docs/references/adcd/autodocs-enriched/VERSION-DIFF.md` showing functions added/changed/deprecated between Autodocs 2.0 (V36) and 3.5 (V39+).

**Condition:** Only build this if Pass 1 inspection reveals that both Autodoc sets use the same library/function naming convention and function entries are matchable by name. If the structures are incompatible, defer to TODOS.md.

**If buildable, output format:**
```markdown
## dos.library

### Added in V39+
- `SetFileDate()` — Set the date on a file
- `MatchPatternNoCase()` — Case-insensitive pattern matching

### Changed (description differs)
- `Open()` — added MODE_READWRITE documentation

### Deprecated
- (none)
```

**How matched:** Compare function name lists between the two Autodoc sets. "Added" = in 3.5 but not 2.0. "Deprecated" = in 2.0 but not 3.5 with DEPRECATED/OBSOLETE tag. "Changed" = in both but with different description text.

### Expansion 5: Struct/Type Definition Index

**What:** `docs/references/adcd/TYPES.md` indexing every struct, typedef, and enum mentioned across all manuals.

**Detection heuristic:** Scan converted markdown for patterns:
- `struct FooBar` (word boundary match, PascalCase or ALL_CAPS)
- Known Amiga type names from include files: Window, Screen, IntuiMessage, FileInfoBlock, etc. Start with a seed list of ~50 common types from the existing autodocs and expand.
- Match against a curated list rather than pure regex to avoid false positives on common words.

**Output format:**
```markdown
## FileInfoBlock

- **Library:** dos.library
- **Documentation:**
  - [Libraries Ch.27: Examining Objects](libraries/27-dos/examine-objects.md)
  - [Autodoc: Examine()](../autodocs/dos.library.md#examine)
  - [Autodoc: ExNext()](../autodocs/dos.library.md#exnext)
```

**Generated in:** Pass 2 enrichment.
