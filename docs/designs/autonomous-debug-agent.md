# Design: Autonomous Amiga Debug Agent — Enforcer + bgdbserver

## Problem Statement

When a ported program crashes on real AmigaOS (running in FS-UAE), Claude gets nothing useful — just a dead emulator or a Guru Meditation screenshot. The debugging loop is completely broken: Claude can't see crash addresses, can't read memory, can't map faults to source lines, and can't autonomously iterate on fixes.

The existing FS-UAE testing pipeline (ADR-014) handles the happy path — run tests, capture TAP results, exit cleanly. But when things go wrong, there's a black hole. A port that crashes requires a human to manually boot FS-UAE, reproduce the crash, run Enforcer, read the hit report, cross-reference the address with a linker map, find the bug, fix it, and repeat. This is the most time-consuming and expertise-dependent part of porting.

## What Makes This Cool

Two audiences, one artifact:

1. **For amiport's porting agents**: Claude can autonomously detect, diagnose, and fix crashes in ported programs. A port that Guru Meditates isn't a dead end anymore — it's just another iteration in the fix-rebuild-test loop. The agent captures the Enforcer hit, maps the address to source line 47 of `ported/main.c`, identifies the NULL pointer dereference, adds the guard, rebuilds, retests, and the port runs clean. No human intervention.

2. **For the Amiga/retro computing community**: An AI agent that can drive Enforcer, parse crash dumps, use GDB over TCP to inspect Amiga memory, and fix bugs in 68k code — that's never been done. The tooling built here (serial capture, Enforcer parser, addr2line integration, GDB automation) is useful beyond amiport.

## Constraints

- Must integrate with the existing ADR-014 FS-UAE testing pipeline (`make test-fsemu`)
- Must work with stock FS-UAE (no patched binaries for the core crash detection layer)
- FS-UAE requires a 68030+ CPU config for MMU emulation (Enforcer needs the MMU). Debug test runs use A1200 with 68030 accelerator card emulation — this differs from normal test runs (stock 68020 A1200). The 68030 data cache could theoretically mask or expose timing-sensitive bugs, but this is acceptable since we're debugging crashes, not performance. Normal (non-debug) test runs continue to use the stock 68020 A1200 config.
- bgdbserver requires bsdsocket.library inside the emulated Amiga (already needed for Category 4 ports)
- Binaries must be compiled with `-g` for STABS debug info and with a linker map for address resolution
- Enforcer, Mungwall, SegTracker, and bgdbserver must be available inside the emulated Amiga filesystem
- Serial → TCP latency is acceptable for debugging (not performance-critical)
- The debug agent must be a Claude skill/agent, not a standalone tool

## Premises

1. The primary debugging need is crash diagnosis (Guru Meditations, illegal memory access, NULL pointers) — not interactive stepping through working code
2. A layered approach is better than picking one tool: Enforcer (passive detection) + bgdbserver (on-demand investigation)
3. Address-to-source mapping via `-g` and addr2line is the critical bridge that makes crash data actionable for Claude
4. bgdbserver's stability issues are manageable for targeted use — Claude only connects GDB when Enforcer finds something, not for long-running sessions

## Approaches Considered

### Approach A: Enforcer Pipeline Only
Always-on crash detection via serial → TCP. Parse Enforcer hits, map to source, fix, rebuild, retest. No interactive debugger.
- Completeness: 6/10 — covers illegal memory access but can't inspect state at crash site

### Approach B: Enforcer + bgdbserver (Chosen)
Everything from A, plus on-demand GDB sessions via bgdbserver for deeper investigation. When passive detection finds a crash location, Claude sets a breakpoint, reruns, inspects registers/memory/locals.
- Completeness: 8/10 — full detect → investigate → fix loop

### Approach C: Enforcer + uae-dap
Everything from A, plus uae-dap patched FS-UAE with embedded GDB server. Most powerful but requires maintaining custom emulator builds. FS-UAE 4.x is not yet stable.
- Completeness: 10/10 but the custom build is an ocean, not a lake

## Recommended Approach

**Approach B: Enforcer + bgdbserver** — layered crash detection and on-demand debugging.

The architecture has two layers that can be adopted independently:

### Layer 1: Passive Crash Detection (Always On)

**Goal:** Every test-fsemu run automatically captures crash data and maps it to source.

#### Components

**1a. FS-UAE Configuration Changes**

Modify the test-fsemu pipeline to enable crash detection:

```ini
# Add to FS-UAE debug test config (separate from normal test config)
amiga_model = A1200                            # Base model
cpu = 68030                                    # 68030 accelerator — enables MMU for Enforcer
serial_port = tcp://127.0.0.1:1234/wait        # Serial → TCP with wait
jit_compiler = 0                               # JIT interferes with debugging
```

The `/wait` flag pauses Amiga boot until the host serial listener connects, ensuring no Enforcer output is lost. **Startup ordering is critical:** the TCP listener must be bound and listening before FS-UAE launches. The serial-capture script polls for port readiness before starting FS-UAE (see 1c below).

**1b. Amiga-Side Debug Tools**

Install in the emulated system (via shared volume or pre-configured HDF):

| Tool | Purpose | Source |
|------|---------|--------|
| SegTracker | Maps addresses to loaded program segments | Aminet `dev/debug/SegTracker` |
| Enforcer | Detects illegal memory access via MMU | Aminet `dev/debug/Enforcer` |
| Mungwall | Fills freed memory with sentinel values, detects use-after-free | Aminet `dev/debug/MungWall` |

Startup-Sequence additions (before the test harness runs):

```
Run >NIL: SegTracker
Run >NIL: Mungwall SERIAL
Run >NIL: Enforcer STACKLINES=4 STACKCHECK DATESTAMP
```

Note: Mungwall's `SERIAL` option routes its output to the serial port alongside Enforcer hits, so use-after-free reports (Mungwall-triggered Enforcer hits) have full context captured by the host listener.

**1c. Host-Side Serial Listener**

A new script `scripts/serial-capture.sh` that:
1. Listens on TCP port 1234 before FS-UAE starts
2. Captures all serial output to a timestamped log file
3. Runs in background alongside FS-UAE
4. When FS-UAE exits, the log is available for parsing

```bash
# Pseudocode — use Python for portability (macOS nc differs from GNU nc)
python3 scripts/serial-capture.py "$PORT_DIR/enforcer.log" &
SERIAL_PID=$!
# Wait for listener to signal readiness via stdout (prints "READY" when bound)
# Do NOT probe-connect — that would consume the listener's accept() slot
read -r READY_LINE < /proc/$SERIAL_PID/fd/1  # or use a named pipe / temp file
# ... start FS-UAE (its /wait will connect to the listener) ...
wait $FSUAE_PID
kill $SERIAL_PID 2>/dev/null

# serial-capture.py uses a READY file sentinel for cross-platform compatibility:
# 1. Binds TCP socket on port 1234
# 2. Writes a sentinel file ($PORT_DIR/.serial-ready)
# 3. Accepts one connection (from FS-UAE)
# 4. Streams all data to enforcer.log
# The launch script polls for the sentinel file, not the TCP port.
```

**1d. Enforcer Hit Parser**

A new script `scripts/parse-enforcer.py` that parses Enforcer output into structured JSON:

```json
{
  "hits": [
    {
      "type": "LONG-READ",
      "address": "0x00000014",
      "illegal": true,
      "pc": "0x00343F4A",
      "usp": "0x00348734",
      "registers": {
        "d0": "0x00000000", "d1": "0x00000005", ...
        "a0": "0x00000000", "a1": "0x00343F00", ...
      },
      "stack_trace": ["0x00343F2E", "0x00343E10", "0x00343D00"],
      "task_name": "Background CLI",
      "segment": {"hunk": 0, "offset": "0x005A"},
      "segtracker": "ports/grep/grep Hunk 0000 Offset 0000005A"
    }
  ],
  "summary": {
    "total_hits": 1,
    "unique_locations": 1,
    "crash_types": ["NULL pointer read (offset 0x14 from NULL struct pointer)"]
  }
}
```

**1e. Address-to-Source Mapper**

A new script `scripts/addr2source.py` that:
1. Takes a hunk number + offset from an Enforcer hit (e.g., Hunk 0, Offset 0x005A)
2. Reads the linker map file (generated at build time with `-Wl,-Map=linker.map`) to find the base address of each hunk. The linker map contains section addresses — hunk N's base address is the `.text`/`.data`/`.bss` section start for that hunk.
3. Computes the absolute link-time address: `base_address_of_hunk_N + offset`. For single-hunk programs (most ports), hunk 0 code base is 0x0 in the bebbo toolchain's a.out intermediate — the offset IS the address and no map parsing is needed. For multi-hunk programs, the linker map is required to find each hunk's base.
4. Runs `m68k-amigaos-addr2line -e <binary> <absolute_address>` to get source file and line number. Note: addr2line works on the a.out/ELF intermediate (before hunk conversion); the `-g` flag preserves STABS debug info through the link stage.
5. Returns structured mapping:

```json
{
  "address": "0x00343F4A",
  "hunk": 0,
  "offset": "0x005A",
  "source_file": "ported/main.c",
  "line": 47,
  "function": "process_input",
  "source_context": [
    "45:   struct node *n = find_node(key);",
    "46:   /* BUG: n can be NULL if key not found */",
    "47:   printf(\"%s\\n\", n->name);  /* <-- CRASH HERE */",
    "48:   return 0;"
  ]
}
```

**1f. Build Changes**

Add to the port Makefile template:
```makefile
# Debug build for crash detection
CFLAGS += -g                          # STABS debug info
LDFLAGS += -Wl,-Map=$(TARGET).map     # Linker map for addr2line
```

The compiled binary with debug info is used for test runs. The strip step moves to packaging only (when building for Aminet release).

**1g. Integration with test-fsemu**

Modify `scripts/test-fsemu.sh` to:
1. Start serial listener before FS-UAE
2. Run Enforcer/Mungwall/SegTracker in the Startup-Sequence
3. After FS-UAE exits, check `enforcer.log`
4. If hits found: parse → map to source → include in TEST-REPORT.md
5. Return structured crash data for the debug agent to act on

**1h. Hard Crash Detection (Guru Meditations)**

Some crashes (illegal instruction, bus error on non-MMU-protected address) trigger a Guru Meditation that resets the machine before Enforcer can report. These are detected by absence: if FS-UAE exits without UAEQuit (no clean exit signal) AND the enforcer.log is empty or incomplete AND the TAP results show no test completion, classify this as a "hard crash."

For hard crashes, the debug agent:
1. Checks FS-UAE's own log for crash indicators (if available)
2. Falls back to bisection: comment out sections of the ported code to isolate which function causes the hard crash
3. Once isolated, add Enforcer breakpoints around that function and rerun

This is a degraded mode — less precise than Enforcer hits but still better than "it crashed, I have no idea why."

**1i. Debug Tools Setup**

New make target: `make setup-debug-tools`

Downloads Enforcer, Mungwall, SegTracker, and bgdbserver from Aminet into `toolchain/debug-tools/`. Integrates with the existing `make setup-toolchain` flow — `setup-toolchain` calls `setup-debug-tools` automatically. A script `toolchain/scripts/setup-debug-tools.sh` handles the downloads, extraction, and placement.

### Layer 2: On-Demand GDB Debugging

**Goal:** When Layer 1 identifies a crash location, Claude can attach GDB to investigate the state at that exact point.

#### Components

**2a. bgdbserver Setup**

Install bgdbserver in the emulated system. It listens on TCP port 2345 inside the Amiga. FS-UAE's bsdsocket emulation routes this to the host.

**VERIFICATION GATE:** Before implementing Layer 2, verify that bgdbserver's TCP port is accessible from the host through FS-UAE's bsdsocket emulation. Test: run bgdbserver inside FS-UAE listening on port 2345, then `telnet localhost 2345` from the host. If this fails, the fallback is to route GDB over the serial port instead (bgdbserver supports serial, and FS-UAE already maps serial → TCP for Layer 1). This would require bgdbserver to use serial instead of its own TCP socket, adding complexity but not blocking Layer 2 entirely.

**2b. GDB Automation Script**

A new script `scripts/gdb-debug.py` that automates m68k-amigaos-gdb:
1. Takes a crash location (source file + line) from Layer 1
2. Generates a GDB command file:
   ```gdb
   target remote :2345
   break ported/main.c:47
   continue
   # When breakpoint hits:
   info locals
   info args
   print n
   print *n
   bt full
   x/16x $sp
   detach
   quit
   ```
3. Runs `m68k-amigaos-gdb -x commands.gdb <binary>`
4. Captures output into structured JSON
5. Handles bgdbserver unresponsiveness with a timeout (kill and retry once)

**2c. Debug Session Flow**

```

                    ┌─────────▼───────────┐
```

### Claude Agent / Skill Design

**New agent: `debug-agent`** (`.claude/agents/debug-agent.md`)

Dispatched automatically when test-fsemu detects Enforcer hits. Capabilities:
- Parse Enforcer hit JSON
- Read source context around crash location
- Determine if the crash is "obvious" or needs GDB investigation using this heuristic:
  - **Obvious (fix directly):** Enforcer address < 0x1000 (NULL pointer + small struct offset), OR address matches known Mungwall sentinel patterns (0xDEADBEEF for freed memory, 0xABADCAFE for pre-fill, 0xCCCCCCCC for uninitialized), OR source context at crash line shows an unchecked pointer dereference, array access without bounds check, or missing NULL return check
  - **Needs GDB:** Enforcer address is in a valid memory range (not NULL, not Mungwall sentinel), OR source context at crash line doesn't reveal an obvious bug, OR multiple Enforcer hits at different locations suggest a deeper issue
- If obvious: fix directly, rebuild, retest
- If subtle: launch GDB session, inspect state, then fix
- **Rollback safety:** Before each fix attempt, the agent copies the current `ported/` directory to a temp backup (`cp -r ported/ /tmp/debug-backup-N/`). If a fix introduces a regression (new crash at a different location, or more Enforcer hits than before), the agent restores from the backup (`cp -r /tmp/debug-backup-N/* ported/`) and tries a different approach. File-copy is used instead of git stash to avoid fragility with stash/pop conflicts and orphaned stashes if the agent is interrupted.
- Maximum 5 fix-rebuild-retest iterations before escalating to user

**New skill: `/debug-amiga`** (`.claude/skills/debug-amiga/SKILL.md`)

User-invokable skill for debugging a specific port:
```
/debug-amiga ports/grep
```
Runs the Enforcer-enabled test, captures crashes, and enters the autonomous fix loop.

### Crash Pattern Knowledge Base

A persistent knowledge base at `docs/references/crash-patterns.md` stores structured crash signature → fix pattern entries. When the debug agent fixes a crash, it appends an entry with: Enforcer signature, root cause, fix applied, and which port it was found in. Over time this becomes a lookup table that accelerates diagnosis. The three common patterns (NULL struct member access, use-after-free via Mungwall sentinels, stack overflow) are seeded initially; new patterns accumulate across ports.

## Open Questions

1. **bgdbserver TCP routing through FS-UAE** — **BLOCKING for Layer 2.** Must verify before implementing Layer 2. Test: run bgdbserver inside FS-UAE, `telnet localhost 2345` from host. Fallback if it fails: route GDB over serial port instead — but this conflicts with Enforcer's serial output. Resolution options: (a) use FS-UAE's second serial port for GDB, (b) stop Enforcer before starting a GDB session, or (c) multiplex protocols on one serial stream.
2. **Debug build size** — How much larger are binaries with `-g`? If significantly larger, we may need separate debug and release build targets.
3. **Enforcer vs MuForce** — MuForce is a more modern rewrite via MuLibs abstraction. Worth evaluating if it provides better output format or additional features. Not blocking — Enforcer is the proven default.
4. **bgdbserver stability** — The reported unresponsiveness may be environment-specific. Test with our exact FS-UAE + bsdsocket setup before committing to Layer 2. The timeout+retry in gdb-debug.py mitigates this.
5. **Timeout interaction** — The existing test-fsemu.sh timeout (30s) must be increased for debug runs. Enforcer adds overhead (MMU checks on every memory access), and serial `/wait` adds startup delay. Debug timeout should be 2-3x normal timeout.

## Success Criteria

- [ ] Enforcer hits are captured automatically during every `make test-fsemu` run
- [ ] Crash addresses are mapped to source file + line number with >90% accuracy
- [ ] The debug agent can autonomously fix at least 3 common crash patterns (NULL deref, stack overflow, use-after-free)
- [ ] A port that crashes on first run reaches a clean test pass within 5 automated fix-rebuild-retest iterations
- [ ] GDB sessions via bgdbserver work for targeted investigation (breakpoint, inspect locals, backtrace)
- [ ] Debug agent prompt includes the three common crash patterns (NULL deref, stack overflow, use-after-free) and successfully applies them
- [ ] Integration requires zero changes to existing ports — the debug pipeline is transparent
