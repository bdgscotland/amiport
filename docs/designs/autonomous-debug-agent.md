# Design: Autonomous Amiga Debug Agent — Enforcer + bgdbserver

Generated by /office-hours on 2026-03-21

Mode: Builder

## Problem Statement

When a ported program crashes on real AmigaOS (running in FS-UAE), Claude gets nothing useful — just a dead emulator or a Guru Meditation screenshot. The debugging loop is completely broken: Claude can't see crash addresses, can't read memory, can't map faults to source lines, and can't autonomously iterate on fixes.

The existing FS-UAE testing pipeline (ADR-014) handles the happy path — run tests, capture TAP results, exit cleanly. But when things go wrong, there's a black hole. A port that crashes requires a human to manually boot FS-UAE, reproduce the crash, run Enforcer, read the hit report, cross-reference the address with a linker map, find the bug, fix it, and repeat. This is the most time-consuming and expertise-dependent part of porting.

## What Makes This Cool

Two audiences, one artifact:

1. **For amiport's porting agents**: Claude can autonomously detect, diagnose, and fix crashes in ported programs. A port that Guru Meditates isn't a dead end anymore — it's just another iteration in the fix-rebuild-test loop. The agent captures the Enforcer hit, maps the address to source line 47 of `ported/main.c`, identifies the NULL pointer dereference, adds the guard, rebuilds, retests, and the port runs clean. No human intervention.

2. **For the Amiga/retro computing community**: An AI agent that can drive Enforcer, parse crash dumps, use GDB over TCP to inspect Amiga memory, and fix bugs in 68k code — that's never been done. The tooling built here (serial capture, Enforcer parser, addr2line integration, GDB automation) is useful beyond amiport.

## Constraints

- Must integrate with the existing ADR-014 FS-UAE testing pipeline (`make test-fsemu`)
- Must work with stock FS-UAE (no patched binaries for the core crash detection layer)
- FS-UAE requires a 68030+ CPU config for MMU emulation (Enforcer needs the MMU). Technically, Enforcer works on 68020+68851, but 68030 is the practical minimum and what FS-UAE emulates. Debug test runs use A1200 with 68030 accelerator card emulation — this differs from normal test runs (stock 68020 A1200). The 68030 data cache could theoretically mask or expose timing-sensitive bugs, but this is acceptable since we're debugging crashes, not performance. Normal (non-debug) test runs continue to use the stock 68020 A1200 config.
- bgdbserver requires bsdsocket.library inside the emulated Amiga (FS-UAE's `bsdsocket_library = 1` must be set explicitly — it is **off by default**)
- Binaries must be compiled with `-gstabs` for STABS debug info (the only recommended debug flag for AmigaOS — `-g`, `-ggdb`, `-gdwarf` may not work correctly) and with a linker map for address resolution
- Enforcer, Mungwall, SegTracker, and bgdbserver must be available inside the emulated Amiga filesystem
- Serial → TCP latency is acceptable for debugging (not performance-critical)
- The debug agent must be a Claude skill/agent, not a standalone tool

## Premises

1. The primary debugging need is crash diagnosis (Guru Meditations, illegal memory access, NULL pointers) — not interactive stepping through working code
2. A layered approach is better than picking one tool: Enforcer (passive detection) + bgdbserver (on-demand investigation)
3. Address-to-source mapping via `-gstabs` and addr2line is the critical bridge that makes crash data actionable for Claude
4. bgdbserver's stability issues are manageable for targeted use — Claude only connects GDB when Enforcer finds something, not for long-running sessions

## Approaches Considered

### Approach A: Enforcer Pipeline Only
Always-on crash detection via serial → TCP. Parse Enforcer hits, map to source, fix, rebuild, retest. No interactive debugger.
- Completeness: 6/10 — covers illegal memory access but can't inspect state at crash site

### Approach B: Enforcer + bgdbserver (Chosen)
Everything from A, plus on-demand GDB sessions via bgdbserver for deeper investigation. When passive detection finds a crash location, Claude sets a breakpoint, reruns, inspects registers/memory/locals.
- Completeness: 8/10 — full detect → investigate → fix loop

### Approach C: Enforcer + uae-dap
Everything from A, plus uae-dap patched FS-UAE with embedded GDB server. Most powerful but requires maintaining custom emulator builds. FS-UAE 4.x is not yet stable.
- Completeness: 10/10 but the custom build is an ocean, not a lake

## Recommended Approach

**Approach B: Enforcer + bgdbserver** — layered crash detection and on-demand debugging.

The architecture has two layers that can be adopted independently:

### Implementation Phasing (Eng Review Decision)

**Layer 1 ships first.** Layer 2 (bgdbserver/GDB) is deferred until the TCP routing verification gate passes (see Open Questions). Layer 1 alone delivers the complete crash detection → source mapping → autonomous fix loop for the most common crash patterns (NULL derefs, use-after-free, stack overflows).

### Layer 1: Passive Crash Detection (Always On)

**Goal:** Every `test-fsemu --debug` run automatically captures crash data and maps it to source.

#### Components

**1a. FS-UAE Configuration Changes**

Debug mode is activated via `test-fsemu.sh --debug`. The existing `generate_config()` function is extended (not duplicated) to add debug-specific settings when the `--debug` flag is passed:

```ini
# Added to FS-UAE config when --debug is active
cpu = 68030                                    # 68030 accelerator — enables MMU for Enforcer
serial_port = tcp://127.0.0.1:<PORT>/wait      # Serial → TCP (ephemeral port)
jit_compiler = 0                               # JIT interferes with debugging
bsdsocket_library = 1                          # Required for bgdbserver (Layer 2)
```

The serial port uses an **ephemeral OS-assigned port** (not a hardcoded port). `serial-capture.py` binds port 0, the OS assigns a free port, and the actual port number is written to a sentinel file. `test-fsemu.sh` reads the sentinel file and generates the FS-UAE config dynamically. This eliminates port conflicts from stale processes or concurrent debug sessions.

The `/wait` flag pauses Amiga boot until the host serial listener connects, ensuring no Enforcer output is lost. **Startup ordering:** serial-capture.py must be bound and its sentinel file written before FS-UAE launches.

Debug mode uses a **90-second timeout** (3x the normal 30s) to account for MMU overhead from Enforcer and serial `/wait` startup delay. The timeout is still overridable via `--timeout N`.

**1b. Amiga-Side Debug Tools**

Install in the emulated system (via shared volume or pre-configured HDF):

| Tool | Purpose | Source |
|------|---------|--------|
| SegTracker | Maps addresses to loaded program segments | Aminet `dev/debug/SegTracker` |
| Enforcer | Detects illegal memory access via MMU | Aminet `dev/debug/Enforcer` |
| Mungwall | Fills freed memory with sentinel values, detects use-after-free | Aminet `dev/debug/MungWall` |

Startup-Sequence additions (before the test harness runs):

```
Run >NIL: SegTracker
Run >NIL: Mungwall
Run >NIL: Enforcer STACKLINES=4 STACKCHECK DATESTAMP
```

**Note on serial output:** Both Enforcer and Mungwall output to the serial port **by default** — there is no `SERIAL` command-line option for either tool. Serial is the hardware default. To redirect output *away* from serial, Enforcer offers `PARALLEL`, `RAWIO`, `FILE`, and `STDIO` options. Mungwall has a separate `mungwall.par` binary for parallel output. Since we want serial output (captured via TCP), the default behavior is exactly what we need — no special flags required.

The `STACKCHECK` option requires SegTracker to be running first — it annotates stack longwords with SegTracker's address-to-segment mapping. This is why SegTracker must start before Enforcer in the Startup-Sequence.

**1c. Host-Side Serial Listener**

A new script `scripts/serial-capture.py` (Python daemon):

```
# Lifecycle:
#   bind(0) → write sentinel(port) → accept(1 conn) → stream to file → EOF → exit
```

1. Binds a TCP socket on port 0 (OS-assigned ephemeral port)
2. Writes the assigned port number to a sentinel file (`$PORT_DIR/.serial-ready`)
3. Accepts one connection (from FS-UAE's serial emulation)
4. Streams all received data to `enforcer.log`
5. Exits when the connection closes (FS-UAE shutdown)

`test-fsemu.sh --debug` polls for the sentinel file (with a 5-second fast-fail timeout — if serial-capture.py fails to start, the user gets an immediate error message instead of waiting for the full 90-second watchdog timeout).

**1d. Enforcer Hit Parser + Address Mapper**

A new script `scripts/debug-report.py` with two subcommands:

**`debug-report.py parse <enforcer.log>`** — Parses Enforcer output into structured JSON:

```
# Enforcer hit format (from Enforcer.doc by Michael Sinz):
#
#   [DATESTAMP line if enabled]
#   {TYPE} {to|from} {ADDRESS}  [data={DATA}]  PC: {PC}
#   USP: {USP} SR: {SR} SW: {SW}  ({flags})  TCB: {TCB}
#   Data: {D0} {D1} {D2} {D3} {D4} {D5} {D6} {D7}
#   Addr: {A0} {A1} {A2} {A3} {A4} {A5} {A6} --------
#   Stck: ... (STACKLINES lines of 8 longwords each)
#   ----> {PC} - "{name}" Hunk {NNNN} Offset {XXXXXXXX}
#   [PC-8: ... (optional, shows code around crash)]
#   Name: "{task}" CLI: "{cli}" Hunk {NNNN} Offset {XXXXXXXX}
#
# Hit types: BYTE-READ, BYTE-WRITE, WORD-READ, WORD-WRITE,
#            LONG-READ, LONG-WRITE, (INST), Alert
# Flags: (U0)=user mode, (F)=Forbid, (D)=Disable, (-)=inactive
```

Output JSON:

```json
{
  "hits": [
    {
      "type": "LONG-READ",
      "address": "0x00000014",
      "pc": "0x00343F4A",
      "usp": "0x00348734",
      "sr": "0x0000",
      "flags": {"user_mode": true, "forbid": false, "disable": false},
      "registers": {
        "d0": "0x00000000", "d1": "0x00000005",
        "a0": "0x00000000", "a1": "0x00343F00"
      },
      "stack": ["0x00000000", "0x07848E1C", "0x00009C40"],
      "segtracker": {"name": "grep", "hunk": 0, "offset": "0x005A"},
      "task_name": "New_Shell",
      "cli_name": "grep",
      "datestamp": "21-Mar-26 18:30:00"
    }
  ],
  "summary": {
    "total_hits": 1,
    "unique_locations": 1,
    "crash_types": ["NULL pointer read (offset 0x14 from NULL struct pointer)"]
  }
}
```

**`debug-report.py map <hits.json> <binary>`** — Maps hunk+offset to source file and line:

1. Takes hunk number + offset from each Enforcer hit
2. For single-hunk programs (most ports): offset IS the address — pass directly to addr2line
3. For multi-hunk programs: reads the linker map file (`<binary>.map`, generated by `-Wl,-Map`) to find each hunk's base address. **Error if multi-hunk binary has no .map file** — prints "multi-hunk binary requires .map file — rebuild with -Wl,-Map" rather than silently guessing wrong
4. Runs `m68k-amigaos-addr2line -e <binary> --section=.text <address>` (for code hunks; uses `--section=.data` or `--section=.bss` for data/bss hunks)
5. If addr2line returns `??:0` (no debug info), reports graceful degradation: "no debug info — rebuild with -gstabs"

**Note on bebbo toolchain:** The bebbo/amiga-gcc linker outputs Amiga hunk format **directly** (patched BFD backend) — there is no ELF intermediate. STABS debug info is stored in HUNK_DEBUG blocks within the hunk executable. `m68k-amigaos-addr2line` reads the hunk binary directly via the patched BFD. No separate `.debug` file is needed as long as the binary hasn't been stripped.

Output JSON:

```json
{
  "address": "0x00343F4A",
  "hunk": 0,
  "offset": "0x005A",
  "source_file": "ported/main.c",
  "line": 47,
  "function": "process_input",
  "source_context": [
    "45:   struct node *n = find_node(key);",
    "46:   /* BUG: n can be NULL if key not found */",
    "47:   printf(\"%s\\n\", n->name);  /* <-- CRASH HERE */",
    "48:   return 0;"
  ]
}
```

**1e. Build Changes**

Add to `ports/common.mk` (default build flags for all ports):

```makefile
CFLAGS += -gstabs                     # STABS debug info (always on)
LDFLAGS += -Wl,-Map=$(TARGET).map     # Linker map for addr2line
```

The `-gstabs` flag is the only debug flag recommended for AmigaOS by the AmigaOS wiki. Other flags (`-g`, `-ggdb`, `-gdwarf`) may not work correctly with the bebbo toolchain's native hunk output.

**All committed binaries include STABS debug info.** This adds a few KB overhead but ensures every binary is always debuggable. The `package` target strips the binary for Aminet distribution:

```makefile
package: $(TARGET)
	m68k-amigaos-strip -o $(TARGET).stripped $(TARGET)
	# ... package $(TARGET).stripped into LHA ...
```

**1f. Integration with test-fsemu**

Extend `scripts/test-fsemu.sh` with a `--debug` flag (no separate script):

1. `--debug` starts `serial-capture.py`, waits for sentinel, generates debug FS-UAE config
2. Adds Enforcer/Mungwall/SegTracker startup to the User-Startup
3. After FS-UAE exits, checks `enforcer.log`
4. If hits found: runs `debug-report.py parse` → `debug-report.py map` → includes in TEST-REPORT.md
5. Returns structured crash data for the debug agent to act on
6. Without `--debug`, test-fsemu.sh behaves exactly as before (no serial, no Enforcer, 30s timeout)

**1g. Hard Crash Detection (Guru Meditations)**

Some crashes (illegal instruction, bus error on non-MMU-protected address) trigger a Guru Meditation that resets the machine before Enforcer can report. These are detected by absence: if FS-UAE exits without UAEQuit (no clean exit signal) AND the enforcer.log is empty or incomplete AND the TAP results show no test completion, classify this as a "hard crash."

For hard crashes, the debug agent uses **printf breadcrumbs**:

1. Inserts `fprintf(stderr, "CHECKPOINT: function_name\n")` at each function entry point in the ported source
2. These print to stderr, which goes to the serial port (already captured by serial-capture.py)
3. The last checkpoint seen before the crash narrows the crash location to a specific function
4. Once narrowed: insert finer-grained checkpoints within that function and rerun
5. After locating the crash site: add Enforcer-aware debugging around that code path

This is a degraded mode — less precise than Enforcer hits but concrete and implementable. Hard crashes (illegal instruction, bus error on non-MMU address) are rare for ported C code — the overwhelmingly common crashes are NULL derefs and use-after-free, which Enforcer catches.

**1h. Debug Tools Setup**

New make target: `make setup-debug-tools`

Downloads Enforcer, Mungwall, and SegTracker from Aminet into `toolchain/debug-tools/`. Integrates with the existing `make setup-toolchain` flow — `setup-toolchain` calls `setup-debug-tools` automatically. A script `toolchain/scripts/setup-debug-tools.sh` handles the downloads, extraction, and placement.

Note: bgdbserver download is included in setup but only used by Layer 2 (deferred).

### Layer 2: On-Demand GDB Debugging (DEFERRED)

**Status:** Deferred until the TCP routing verification gate passes. See Open Questions #1.

**Goal:** When Layer 1 identifies a crash location, Claude can attach GDB to investigate the state at that exact point.

**Prerequisites before implementation:**
1. Verify bgdbserver TCP routing through FS-UAE bsdsocket emulation (BLOCKING)
2. Build `m68k-amigaos-gdb` in the Docker toolchain image (requires newer bison)

#### Components (for future implementation)

**2a. bgdbserver Setup**

Install bgdbserver in the emulated system. It listens on TCP inside the Amiga. FS-UAE's bsdsocket emulation uses real host sockets (`bsdsocket_library = 1`), so the Amiga-side TCP port should be accessible from the host.

**VERIFICATION GATE:** Before implementing, verify with: run bgdbserver inside FS-UAE listening on a TCP port, then `telnet localhost <port>` from the host. Research confirmed that FS-UAE bsdsocket calls real POSIX `bind()`/`listen()` — this *should* work but needs empirical verification.

**Important:** bgdbserver is TCP-only (no serial support). If TCP routing fails, the alternative is `astub` (a different GDB stub that works over serial), not routing bgdbserver over serial.

**2b. GDB Automation Script**

A new script `scripts/gdb-debug.py` (not built until Layer 2 is unblocked):
1. Takes a crash location (source file + line) from Layer 1
2. Generates a GDB command file
3. Runs `m68k-amigaos-gdb -x commands.gdb <binary>`
4. Captures output into structured JSON
5. Handles bgdbserver unresponsiveness with a timeout (kill and retry once)

### Claude Agent / Skill Design

**New agent: `debug-agent`** (`.claude/agents/debug-agent.md`)

Dispatched automatically when test-fsemu detects Enforcer hits. Capabilities:
- Parse Enforcer hit JSON
- Read source context around crash location
- Determine if the crash is "obvious" or needs deeper investigation using this heuristic:
  - **Obvious (fix directly):** Enforcer address < 0x1000 (NULL pointer + small struct offset), OR address matches known Mungwall sentinel patterns (0xDEADBEEF for freed memory, 0xABADCAFE for pre-fill, 0xCCCCCCCC for uninitialized), OR source context at crash line shows an unchecked pointer dereference, array access without bounds check, or missing NULL return check
  - **Needs investigation:** Enforcer address is in a valid memory range (not NULL, not Mungwall sentinel), OR source context at crash line doesn't reveal an obvious bug, OR multiple Enforcer hits at different locations suggest a deeper issue. Without Layer 2 (GDB), the agent uses source-level reasoning + crash pattern KB lookup for subtle crashes.
- If obvious: fix directly, rebuild, retest
- If subtle (without Layer 2): consult crash patterns KB, reason about source context, apply best-guess fix
- **Rollback safety:** Before each fix attempt, the agent creates a named git stash (`git stash push -m 'debug-attempt-N'`). If a fix introduces a regression (new crash at a different location, or more Enforcer hits than before), the agent pops the stash (`git stash pop`) and tries a different approach. Named stashes are visible via `git stash list`, discoverable if the agent is interrupted, and survive reboots.
- Maximum 5 fix-rebuild-retest iterations before escalating to user

**New skill: `/debug-amiga`** (`.claude/skills/debug-amiga/SKILL.md`)

User-invokable skill for debugging a specific port:
```
/debug-amiga ports/grep
```
Runs the Enforcer-enabled test (`test-fsemu.sh --debug`), captures crashes, and enters the autonomous fix loop.

### Crash Pattern Knowledge Base

A persistent knowledge base at `docs/references/crash-patterns.md` stores structured crash signature → fix pattern entries. When the debug agent fixes a crash, it appends an entry with: Enforcer signature, root cause, fix applied, and which port it was found in. Over time this becomes a lookup table that accelerates diagnosis. The three common patterns (NULL struct member access, use-after-free via Mungwall sentinels, stack overflow) are seeded initially; new patterns accumulate across ports.

## Open Questions

1. **bgdbserver TCP routing through FS-UAE** — **BLOCKING for Layer 2 only.** Must verify before implementing Layer 2. Research confirmed FS-UAE bsdsocket uses real host sockets (POSIX `bind()`/`listen()`), so this should work — but needs empirical verification with bgdbserver specifically. **Note:** bgdbserver is TCP-only. If TCP routing fails, the alternative is `astub` (a serial-based GDB stub), not "bgdbserver over serial."
2. **Debug build size** — `-gstabs` adds STABS data in HUNK_DEBUG blocks. Expected overhead is a few KB for typical CLI ports. Since all committed binaries now include debug info (stripped only at package time), monitor repo size. If it becomes a problem, add a `make release` target.
3. **Enforcer vs MuForce** — MuForce is a more modern rewrite via MuLibs abstraction. Worth evaluating if it provides better output format or additional features. Not blocking — Enforcer is the proven default.
4. **bgdbserver stability** — The reported unresponsiveness may be environment-specific. Test with our exact FS-UAE + bsdsocket setup before committing to Layer 2. The timeout+retry in gdb-debug.py mitigates this.
5. ~~**Timeout interaction**~~ — **Resolved:** Debug mode defaults to 90s timeout (3x normal), overridable via `--timeout`.
