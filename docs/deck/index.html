The question for this room: what happens when we apply this approach to the migration engagements we sell to clients?
The chatbot approach to code migration fails for four specific reasons.
The playbook most teams start with: paste legacy code into an LLM. Ask it to convert to Java. Get something that compiles. Ship it. Discover in testing that the business logic is subtly wrong — and you have no way to trace why.
Prompt: Convert this COBOL to Java.

AI: import com.ibm.cics.CICSAdapter;
  // This class does not exist.
  // The AI invented it to be helpful.
  // Nobody catches it until UAT.
This isn't hypothetical. In the system I built, the target platform's getopt_long() function exists — it links, it compiles — but it returns '?' for every single valid option. Every program silently fails. You don't catch it without a test suite that actually runs on the target.
NO CLASSIFICATION
Every API call gets the same treatment. stat() has a direct equivalent — automate it. fork() fundamentally doesn't exist — it needs human architecture. The chatbot doesn't distinguish between a 5-minute rename and a 5-week redesign.
NO VERIFICATION
The output isn't compiled. Isn't tested. Isn't run on the target platform. "It looks right" is the only quality gate. We shipped grep to a public archive and it had exit codes always returning 0 — a bug invisible to anything except a real test suite.
NO PLATFORM KNOWLEDGE
The LLM doesn't have the target platform's API documentation, its C runtime quirks, or the 19 crash patterns we've documented. It doesn't know that vsnprintf(NULL, 0) crashes the runtime, or that fclose(stdin) takes down the entire operating system desktop.
NO LEARNING
Module 5,000 hits the exact same 68k alignment bug that module 12 hit. Every conversation starts from zero. There's no mechanism for one migration to make the next one better.
The stress test: modern Linux to a 1992 Commodore Amiga.
Two platforms that share zero APIs. Different CPU architecture. Different memory model. Different everything. If the approach works here — 2MB RAM, 7MHz, no memory protection — the patterns transfer to enterprise targets where the gap is smaller and the tooling is better.
SOURCE: POSIX/LINUX 2024
64GB+ RAM, virtual memory, protected processes
Automatic process cleanup on exit
open() / read() / stat() / regex / fork()
Threads, sockets, dynamic loading built in
TARGET: AMIGAOS 1992
2MB RAM, 7MHz Motorola 68000
No memory protection — one bad pointer corrupts the OS
No process cleanup — leaked memory persists until reboot
No regex, no threads, no fork(), no sockets
Dimension-by-Dimension Gap
File I/Oopen()→Open() different flags, semantics
Metadatastat()→Lock()+Examine() via BCPL pointers
Epoch1970-01-01→1978-01-01 252M second offset
Exit codes0/1→0/5/10/20 exit(1) is invisible
SignalsSIGINT via kernel→Must poll SetSignal() manually
Stack8MB+ extensible→4KB default, fixed, no guard pages
MemoryVirtual, per-process cleanup→Flat physical, no cleanup

7-STAGE PIPELINE WITH 3 FEEDBACK LOOPS
LOOP 1: BUILD FAILS → TRANSFORM
Missing header, undefined symbol, type mismatch — routes back to the transform stage. If a shim function is missing, /extend-shim writes one. Up to 5 iterations.
LOOP 2: CRASH → DEBUG AGENT → BUILD
Guru Meditation on FS-UAE → debug-agent auto-dispatched. Parses Enforcer hit data, maps crash address to source line via addr2line, queries the 19-pattern crash KB, applies fix with git stash rollback safety, rebuilds, retests. Up to 5 iterations, no human.
LOOP 3: REVIEW CHANGES → BUILD
Memory-checker finds a leak or perf-optimizer changes a hot loop → code goes back through build and test. Every change is re-verified. The grep re-port: fgetc()→fgets() was a 3-5x speedup on the 68000.
Enterprise: automated regression, rollback, and fix loops. Not a waterfall that fails silently in UAT.

When the system gets it wrong, it marks its own mistakes.
Every migration system makes mistakes. The question is whether it buries them or learns from them. Three real stories.
THE DEBUNKED PATTERN
We spent weeks building workarounds for exit() "hanging" on AmigaOS. Restructured cleanup code across all 14 ports. Built a custom _exit() pattern that bypassed atexit handlers.
The actual cause: a single non-ASCII byte in a test script. ARexx — a 1987 scripting language — crashes on any UTF-8 character, including inside comments. The \= not-equal operator isn't recognized either — must use ~=.
Pattern #9 is now marked DEBUNKED in the knowledge base with a full post-mortem. The _exit() pattern was retired. exit() with atexit() cleanup was restored as the correct approach.
The system corrected its own mistake and updated its rules.
THE GREP RE-PORT
grep shipped to Aminet. Real users downloaded it. Two critical bugs discovered post-publication:
BUG-1: Exit codes always 0. Missing braces in an if/else block — exit() executed unconditionally. Shell scripts using IF WARN got wrong results.
BUG-2: grep -R silently skipped subdirectories. d_type == DT_DIR check — but AmigaOS never populates d_type. Always DT_UNKNOWN.
Both patterns added to known-pitfalls. Source-analyzer now flags braceless multi-statement if/else AND d_type checks. Caught at analysis for all future ports.
THE DESKTOP KILLER
A ported program called fclose(stdin) after processing input. On Linux, harmless. On AmigaOS, it closes the shell's console handler and crashes the entire Workbench desktop. Not just the program — the operating system. User reboots.
Now in known-pitfalls. Source-analyzer flags every fclose() that could receive stdin. Caught at analysis, before any code is written.
