5. UNFIXED: Math Library Function Name strdups (6× critical leak)
   Location: main.c:343-348 in open_new_file()
   Problem: lookup(strdup("e"), FUNCT) and 5 others — no tracking or cleanup
   Impact: 6× 2-byte strings = 12 bytes per "bc -l" invocation
   Root Cause: Strings allocated but pointers stored inside lookup()'s internal data,
     not returned for caller to track. bc_cleanup() doesn't iterate lookup()
     structures to free individual allocated strings.
   Why Not Fixed: The first pass audit identified this, but the fix wasn't applied.
     The strdups are still in open_new_file() at lines 343-348, with NO
     corresponding free calls in bc_cleanup().
   How to Fix: Track strdups in an array (math_strdups[MAX_MATH_STRDUPS]) and
     free them in bc_cleanup(). See detailed fix below.

6. UNFIXED: Global Storage Array Name Entries (moderate-to-critical leak)
   Location: storage.c allocation, cleanup at main.c:60-68
   Problem: bc_cleanup() frees f_names, v_names, a_names ARRAYS but NOT the
     individual string pointers STORED IN each array. Each entry was allocated
     via lookup()->strcopyof() and should be freed.
   Code Issue:
     Line 61:  if (f_names[0]) free(f_names[0]); /* Only frees f_names[0] */
     Line 62:  free(f_names);  /* Frees ARRAY, not entries [1..n] */
   Impact: 10-100+ bytes per invocation (depends on function/variable/array count)
   Root Cause: The cleanup function partially addresses this (frees f_names[0] the
     "(main)" entry) but leaves f_names[1..f_count-1] with live malloc'd pointers.
   How to Fix: Iterate and free each entry before freeing the array:
     for (i = 0; i < f_count; i++)
       if (f_names[i]) free(f_names[i]);
     free(f_names);
   Same pattern needed for v_names[0..v_count-1] and a_names[0..a_count-1].

7. UNFIXED: Function Body Buffers (moderate leak)
   Location: storage.c:105 allocations, cleanup missing
   Problem: Each bc_function has an f_body buffer allocated at startup
     (via more_functions() -> bc_malloc()). These are never freed at exit.
   Impact: 50-200+ bytes per invocation (depends on function count)
   Root Cause: The bc_cleanup() function frees the functions ARRAY but not the
     contents of each function struct. Each functions[i].f_body is malloc'd but
     never freed.
   How to Fix: In bc_cleanup(), before freeing functions array:
     for (i = 0; i < f_count; i++)
       if (functions[i].f_body) free(functions[i].f_body);
     free(functions);

8. UNFIXED: Global Constants _zero_, _one_, _two_ (critical leak)
   Location: number.c:55-57 (globals), number.c:109-113 (initialization),
     cleanup missing
   Problem: These reference-counted bc_num objects are allocated at startup and
     never freed. The reference-counting system prevents normal bc_free_num()
     cleanup because these globals are copied many times throughout execution,
     leaving n_refs > 1 at exit.
   Code Issue:
     Line 109: _zero_ = bc_new_num(1, 0);  /* Allocates bc_struct + data */
     /* Never freed despite bc_cleanup() being called */
   bc_free_num() logic (number.c:90-101):
     (*num)->n_refs--;
     if ((*num)->n_refs == 0) { free data }  /* Only frees if n_refs==0 */
   Impact: 40 bytes per constant × 3 = 120 bytes minimum, plus any intermediate
     bc_num objects still referencing these
   Root Cause: bc_cleanup() was added but no code to handle reference-counted
     objects. Can't just call bc_free_num() — would need to reset n_refs first.
   How to Fix: Force-free in bc_cleanup():
     if (_zero_ != NULL) {
       if (_zero_->n_ptr) free(_zero_->n_ptr);
       free(_zero_);
     }
     /* Same for _one_, _two_ */

9. UNFIXED: _bc_Free_list Cache (critical variable leak)
   Location: number.c:59 (static cache), cleanup missing
   Problem: bc_free_num() maintains a linked list cache of freed bc_num objects
     for reuse. This cache is never walked and freed at exit.
   Code Issue:
     Line 97-98 (in bc_free_num):
       (*num)->n_next = _bc_Free_list;
       _bc_Free_list = *num;  /* Freed nodes go to cache */
     /* Cache never freed at exit */
   Impact: Variable, depends on program. Could be 10-1000+ bytes for complex
     bc programs (each cached node is ~40 bytes + 1-100 bytes data buffer)
   Root Cause: The cache is an optimization detail in number.c, not exposed
     to main.c cleanup code. No mechanism to cleanup the cache.
   How to Fix: Either add cleanup function to number.c or make _bc_Free_list
     accessible to main.c. Then in bc_cleanup():
     while (_bc_Free_list != NULL) {
       node = _bc_Free_list;
       _bc_Free_list = node->n_next;
       if (node->n_ptr) free(node->n_ptr);
       free(node);
     }
